## 1. 项目目标与痛点

**痛点**：游客在景点面对一座雕像或一栋老建筑时，通常只能看见一块字数有限、文绉绉的介绍牌。无法对话、无法深入、无法跟随这片土地真正的"人"去理解它。

**解决方案**：让真实的历史人物本人作为 AI 角色，**亲自向你讲述他生活过的城市**。  
苏轼讲西湖，鲁迅讲鲁迅公园 —— 由 Gemma 4 完成从"识别 → 锁定人物 → 角色扮演"的全链路。

---

## 2. 模型选型理由

### 2.1 视觉感知：Gemma 4 31B IT (Dense)

**为什么不是 Gemma 4 26B A4B**：MoE 模型在多模态稀疏激活下，对细节（碑文小字、雕像面部特征）的识别一致性不如 Dense 模型。

**为什么不是 Gemini 2.5 Flash**：
1. 大赛主题是 Gemma 4 的多模态能力展示，应让 Gemma 担任主角而非配角。
2. Gemini 2.5 Flash 在 Free tier 仅 20 次/天，不足以支撑现场演示。
3. Gemma 4 31B 的 256K 长上下文允许我们一次性传入完整人物清单（30 人）+ 景点清单（90 个）+ 场景类型枚举，进行**单次推理多任务并行**。

**保留 Gemini Flash 作为故障保险**：在 Gemma 4 31B 不可达（罕见的 timeout / region 故障）时仍可降级，避免演示翻车。

### 2.2 历史人物对话：Gemma 4 26B A4B IT (MoE)

**为什么选 MoE**：
- 30 位人物的角色扮演 → 长尾高并发场景
- MoE 单次推理仅激活 4B 参数 → 流式首字延迟低
- 26B 总参数量保证了人物语言风格（古汉语 / 民国白话 / 西南官话）的稳定性

### 2.3 一句话总结

> **Dense 用于「看懂世界」，MoE 用于「持续表达」** —— Gemma 4 家族的双形态特性恰好对应"识别"与"对话"两类异质负载，是本项目选型的核心动机。

## 3. 多模态识别架构

### 3.1 单次推理完成四个子任务

传统做法是串行调用四个模型（OCR → 分类 → 雕像识别 → 检索）。本项目利用 Gemma 4 31B 的 256K 上下文，将所有任务**编码进一次 prompt**：

```
prompt = scene_type_enum + subject_type_enum + character_closed_list(30人) + visual_hints
image = user_upload
↓
Gemma 4 31B 单次推理输出 JSON：
{
  "text_on_image": ["1881-1936"],          // OCR
  "scene_type": "纪念广场",                  // 10 类场景分类
  "subject_type": "人物雕像",                // 6 类主体分类
  "person_in_statue": "鲁迅",                // 30 选 1 人物识别
  "notable_features": ["蓄短发八字胡", "圆框眼镜"],
  "possible_name": "鲁迅公园"
}
```

**收益**：
- API 调用次数：4 → 1（成本与延迟均降低 ~75%）
- 任务一致性：所有判断基于同一视觉理解，避免串行模型的累积误差

### 3.2 多信号加权评分（解决"故居 vs 公园"难题）

**问题**：鲁迅故居和鲁迅公园都属于鲁迅，单靠人名识别会混淆。

**早期失败方案**：级联 first-match 规则（OCR 命中 → 人物匹配 → 场景兜底）。结果：上海鲁迅公园被识别成鲁迅故居（因为人物相同时优先级混乱）。

**当前方案**：对每个候选景点计算综合得分：

| 信号 | 分值 | 说明 |
|------|------|------|
| OCR 景点名直接命中 | **+10** | 例如照片有"鲁迅公园"四个字 |
| 别名命中（如"西子湖" → 西湖） | +3 ~ +6 | 按渠道权重 |
| Gemma 识别出雕像人物 | **+12** | 该人物拥有的所有景点全部加分 |
| `possible_name` 命中 | +3 | Gemma 给出的猜测 |
| `notable_features` 命中 | +2 | 视觉特征 |
| 场景类型匹配 | **+4** / **−2** | 鲁迅公园声明为 `["开阔公园", "园林"]` |
| 主体 × 景点类型冲突 | **−8 (决定性)** | "人物雕像" × "故居" → 否决（故居室内不会有大型雕像） |

**关键创新**：`subjectTypeBonus()` 函数将"主体类型"与"景点类型"做语义冲突检测，例如：
- `subject_type = "人物雕像"` + `landmark.type = "故居"` → **−8 决定性否决**
- `subject_type = "人物雕像"` + `landmark.type = "公园"` → +4
- `subject_type = "建筑外观"` + `landmark.type = "故居"` → +4

这一步彻底解决了"故居 vs 公园"混淆问题。

### 3.3 数据模型扩展

为了支持 3.2 的评分，我们给每个景点增加两个语义字段：

```typescript
export interface Landmark {
  name: string;
  description: string;
  type: LandmarkType;      // 11 类：故居 / 纪念馆 / 公园 / 陵园 / 广场 / 园林 / 古镇 / 校园 / 历史建筑 / 自然景观 / 城市地标
  sceneTypes: SceneType[]; // 10 类：室内陈列 / 民居院落 / 开阔公园 / 纪念广场 / 园林 / 自然景观 / 城市街景 / 历史建筑 / 校园 / 古镇街区
}
```

`type` 和 `sceneTypes` 的双重声明，让一个"人物雕像"主体能精准对应到该人物的**公园**而非**故居**。

## 4. 对话子系统

### 4.1 角色扮演

每位历史人物有独立的 `Persona`：
- 朝代 / 称谓 / 生平
- 语言风格（古汉语 / 民国白话 / 当地方言）
- 守护景点清单
- 全局约束（不出戏 / 不混用现代术语 / 不离开自己的时代）

### 4.2 流式输出

使用 SSE（Server-Sent Events）将 Gemma 4 26B A4B 的输出分片返回，前端逐字渲染。

### 4.3 实用查询挂钩

对话中如检测到关键词（交通 / 门票 / 美食 / 住宿），后端会附加高德地图 deep-link，让历史人物的话术与现实工具自然衔接：

> 鲁迅："实事求是，不妨直接查看 → 高德地图·查看路线"

### 4.4 语音输入

Chat 页集成 Web Speech API，支持中文语音转文字，凑齐"视觉 + 文本 + 音频"三模态。

## 5. 工程亮点

| 维度 | 实现 |
|------|------|
| **状态管理** | React 19 + URL state（无重型状态库） |
| **聊天记录** | IndexedDB 本地持久化（隐私优先） |
| **图片上传** | 客户端压缩（5MB 上限）→ 减少 base64 传输 |
| **错误处理** | 配额识别 / timeout 容错 / SSE 解析降级 |
| **可观测性** | 服务端日志带 `[Scan]` 前缀，便于 Vercel 日志过滤 |
| **安全** | API key 仅服务端读取，前端零暴露；CSP 默认严格 |

## 6. 局限与未来工作

| 当前局限 | 计划 |
|---------|------|
| 城市数量限于 10 座、人物 30 位 | 计划接入更广覆盖（西安、洛阳、广州…） |
